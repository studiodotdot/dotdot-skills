# 교육 과정 설계 (edu-designer)

> 니즈 분석 → 모델 선택·설계 → 요약 테이블 검토 → 설계서 저장 → (선택) HTML 변환

## WHEN TRIGGERED - EXECUTE IMMEDIATELY

이 문서는 실행 지시서다. 사용자의 요청을 받으면 Step 1부터 시작한다.
단, 요청에 A~D 항목 정보가 대부분 담겨 있으면(완성된 브리프) **인터뷰를 생략하고 Step 2로 직행**한다 —
이때 스스로 가정한 항목을 목록으로 명시한다.

---

## 워크플로우

### Step 1: 니즈 분석 인터뷰

교육 설계의 품질은 니즈 분석의 깊이에 비례한다. 사용자의 요청에서 이미 파악된 정보는 확인만 하고,
부족한 정보를 체계적으로 질문한다.

**파악해야 할 항목:**

#### A. 기본 정보
| 항목 | 질문 | 예시 |
|------|------|------|
| 교육 주제 | 어떤 주제의 교육인가요? | AI 활용, 고객 응대, 리더십 등 |
| 교육 배경 | 이 교육을 하게 된 배경/이유가 뭔가요? | 경영진 지시, 현장 니즈, 신규 서비스 런칭 등 |
| 총 시간 | 교육 시간이 정해져 있나요? | 2시간, 4시간, 2일(16시간) 등 |
| 교육 형태 | 어떤 형태로 진행하나요? | 오프라인 집합, 온라인 실시간, 영상, 블렌디드 |

#### B. 학습자 분석
| 항목 | 질문 | 예시 |
|------|------|------|
| 대상 | 누가 교육 받나요? (직군, 직급, 인원수) | 소매 매장 직원 30명, 본사 팀장급 15명 |
| 현재 수준 | 이 주제에 대해 지금 얼마나 알고 있나요? | 완전 초보, 기본은 앎, 수준 천차만별 |
| 학습 동기 | 학습자들이 자발적? 의무적? | 의무 교육, 자발적 신청, 혼재 |

#### C. 목표 분석
| 항목 | 질문 | 예시 |
|------|------|------|
| 기대 행동 | 교육 후 학습자가 뭘 할 수 있으면 되나요? | 도구를 설치해서 쓸 수 있다, 고객 전화를 잘 건다 |
| 현업 적용 | 배운 걸 현업에서 바로 쓸 수 있어야 하나요? | 교육 다음 날부터 적용, 인식 변화만 되면 OK |
| 평가 여부 | 교육 후 평가나 과제가 있나요? | 시험, 실습 과제, 만족도 설문만, 없음 |

#### D. 환경/제약
| 항목 | 질문 | 예시 |
|------|------|------|
| 실습 환경 | 실습이 필요하다면 환경이 갖춰져 있나요? | 개인 노트북, 공용 PC, 실습 불가 |
| 기존 자료 | 참고할 기존 교육 자료나 교안이 있나요? | 있음(파일 제공), 없음(신규) |
| 제약사항 | 반드시 포함/제외해야 할 내용이 있나요? | 특정 도구 필수, 보안상 데모 불가 등 |

**인터뷰 진행 방식:**
1. 사용자의 요청에서 이미 언급된 정보를 먼저 정리한다
2. 정리한 내용을 보여주고, 빠진 항목들을 한 번에 질문한다
3. 질문은 자연스러운 대화체로, 항목별로 묶어서 한다 (테이블 형식으로 묻지 않는다)
4. 사용자가 "모르겠어요" 또는 답을 안 하는 항목은 합리적으로 가정하고 가정한 내용을 명시한다
5. 인터뷰는 최대 2회 왕복으로 마무리한다 (질질 끌지 않는다)
6. 질문이 불가능한 상황(제안서 초안 단계, 자동 실행 등)이면 전 항목을 합리적으로 가정하고
   가정 목록을 산출물에 포함한다

---

### Step 2: 교육 모델 선택 + 커리큘럼 설계

**모델 선택 기준**:

| 상황 | 모델 | 레퍼런스 |
|------|------|----------|
| 기본값 — 정규/조직 교육, 승인·보고 필요, 반복 운영 | ADDIE | `references/addie-model.md` |
| 실습·행동 변화 중심의 단기 실무 교육 ("할 줄 알아야 하는" 교육) | 메릴 | `references/merrill-principles.md` |
| 대규모·다단계 과정 체계, 하위 기능 위계 분석이 중요한 직무 교육 | 딕앤케리 | `references/dick-carey-model.md` |

**기본 조합**: 전체 구조는 ADDIE로 잡고, 각 모듈 내부 차시는 메릴 5단계(문제→활성화→시연→적용→통합)로
설계한다 — 출력 템플릿의 "차시 구성"이 이 조합을 전제로 한다.
레퍼런스는 **기본 조합이면 `addie-model.md`와 `merrill-principles.md` 둘 다**, 단일 모델로 갈 때만
해당 문서 하나를 읽는다 (선택하지 않은 모델 문서는 읽지 않는다).

**항상 읽는 레퍼런스:**
- `references/bloom-taxonomy.md` — 학습목표 수준 분류 + 행동 동사
- `references/teaching-methods.md` — 교수방법 선택 기준 (Bloom 수준 × 인원 × 온/오프라인 매트릭스)

**설계 원칙:**
- 구조화: 모듈 → 주제 → 세부내용의 계층 구조
- 계열화: Easy→Hard, Concrete→Abstract, Known→Unknown 순서
- 모듈 간 연계: 앞 모듈의 내용이 다음 모듈의 기반이 되도록
- 시간 현실성: 모듈당 30분~2시간, 총 시간 합계 일치
- 운영 리듬: 도입 오리엔테이션(10~15분)과 마무리 정리/Q&A를 배분한다.
  2시간 초과 과정은 모듈 경계에 10분 휴식을 배치한다 — 산식이 아니라 배치 가이드
  (예: 4시간 집합 과정이면 2회 내외). 휴식 포함 여부를 총 시간에 명시
- 교수방법: 학습목표 수준에 맞는 방법 선택 (Bloom 수준 ↔ 교수방법 매칭)
- 학습자 수준 반영: 수준 격차가 크면 기초부터 단계적으로, 균일하면 바로 핵심으로

---

### Step 3: 요약 테이블 출력 (검토·수정 루프용)

설계된 커리큘럼을 아래 테이블 형식으로 먼저 보여주고 사용자 확인을 받는다.

| 모듈 | 주제 | 내용 | 세부내용 | 시간 | 교수법 | 비고 |
|------|------|------|----------|------|--------|------|

- **모듈**: M1, M2, M3... 또는 Day1-오전, Day1-오후 등
- **내용**: 학습목표 (Bloom 행동 동사 기반, "~할 수 있다" 형식)
- **세부내용**: 다루는 구체적 항목들 (줄바꿈으로 구분)
- **시간**: 분 단위. 마지막 행에 **합계를 표기하고 총 교육 시간과 일치하는지 검산**한다
- **교수법**: 강의/실습/토론/사례연구/롤플레이/워크숍 등

테이블 아래에 간단한 설계 근거를 덧붙인다:
- 적용한 교육 모델 + 계열화 논리 (왜 이 순서인지 1-2줄)
- 니즈 분석에서 파악한 핵심 포인트가 설계에 어떻게 반영되었는지
- (인터뷰 생략 시) 가정 목록

---

### Step 4: 설계서 저장

사용자가 확인하면 **`references/curriculum-template.md`를 읽고 그 형식대로** 풀 설계서를
Markdown 파일로 저장한다. 템플릿은 과정 개요 / 학습목표 / 모듈 구조도 / 모듈별 상세(차시 구성 포함) /
설계 근거 / 평가 계획 / 시간 배분 요약으로 구성된 결재·제안용 문서다.

- **파일명**: `{교육과정명}_커리큘럼_{YYMMDD}.md`
- **저장 위치**: 사용자의 현재 작업 디렉토리
- 사용자가 "간단하게"를 원하면 과정 개요 + 요약 테이블 + 설계 근거로 축약한다

---

### Step 5: HTML 교육자료 변환 (선택)

사용자가 "HTML로 만들어줘", "슬라이드로 만들어줘" 등을 요청하면 실행한다.

- **vanilla-slide 스킬을 호출**하여 커리큘럼 내용을 HTML 프레젠테이션으로 변환
- 각 모듈/주제를 슬라이드로 분리, 세부내용은 빌드 효과(data-step)로 순차 표시
- 교수법에 맞는 활동 안내 슬라이드 포함

이 단계는 사용자가 명시적으로 요청할 때만 실행한다. 자동 실행하지 않는다.

---

## 수정 지원

사용자가 수정을 요청하면 모듈 단위로 수정한다:
- "M2 빼줘" → 해당 모듈 제거 + 시간 재배분
- "실습 비중 높여줘" → 교수법 조정 + 시간 재배분 (메릴 원리 강화)
- "순서 바꿔줘" → 계열화 재조정
- "모듈 추가해줘" → 새 모듈 삽입 + 전체 시간 재계산

수정 후 요약 테이블을 다시 출력하고, 저장본이 있으면 함께 갱신한다.
